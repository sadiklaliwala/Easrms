1.  Introduction

1.1  Purpose

This document defines the software requirements for the Employee Asset and Service Request Management System. It serves as the single reference for development, testing, and evaluation of the system.

1.2  Project Overview

The system is an internal enterprise portal where employees can raise asset and service requests, managers can approve eligible requests, admins can manage categories and assignments, and support users can resolve assigned requests.

1.3  Scope

In Scope

Authentication simulation with JWT in HttpOnly cookies

Role-based navigation and access control

Request creation, listing, detail view

Category management by admin

Approval flow by manager

Support assignment by admin

Status tracking through 8 defined statuses

Dashboard with role-based metrics

Audit trail and status history

Search, filter, and pagination on listings

Change Request implementation on Day 8

Out of Scope

Real SSO integration

Production email or SMS notifications

File storage in cloud

Complex reporting engine

Background workers (optional only)

User self-registration

1.4  Intended Audience

Interns, Evaluators, Senior Tech Lead, Project Coordinator

3.  Assumptions and Dependencies

JWT will be stored in HttpOnly cookies, not localStorage

Seeded users will be used for login simulation, no registration screen

File attachments are placeholder only, no actual file upload in scope

Email and SMS notifications are out of scope

All users are pre-created by Admin, no self-registration

SQL Server is available as local instance during development

Manager-employee relationship is defined by ManagerId in Users table

Priority values are static and hardcoded on frontend: High, Medium, Low

Status values are static and hardcoded on frontend

Role values are static and hardcoded on frontend

Role-based API authorization, JWT in HttpOnly cookies, no sensitive data exposure

Logging

Testing

9.  Database Schema

Table

Columns

Users

UserId (PK), FullName, Email, PasswordHash, RoleId (FK), ManagerId (FK self-reference), IsActive, CreatedOn, UpdatedOn, LastLoginOn

Roles

RoleId (PK), RoleName

RequestCategory

CategoryId (PK), CategoryName, IsApprovalRequired, IsActive, CreatedOn, UpdatedOn

ServiceRequest

RequestId (PK), RequestNumber, EmployeeId (FK), CategoryId (FK), Title, Description, Priority, Status, AssignedTo (FK), CreatedOn, UpdatedOn, ResolvedOn, ClosedOn, ClosedBy (FK), RejectionReason

RequestComment

CommentId (PK), RequestId (FK), CommentBy (FK), CommentText, CommentType, CreatedOn, IsDeleted

RequestStatusHistory

HistoryId (PK), RequestId (FK), OldStatus, NewStatus, ChangedBy (FK), ChangedOn, Remarks

Table Relationships

Roles → Users (one role has many users)

Users → Users (self-reference, ManagerId)

Users → ServiceRequest (one employee creates many requests)

Users → ServiceRequest (one support user assigned many requests)

RequestCategory → ServiceRequest (one category has many requests)

ServiceRequest → RequestComment (one request has many comments)

ServiceRequest → RequestStatusHistory (one request has many history records)

Users → RequestComment (one user adds many comments)

Users → RequestStatusHistory (one user triggers many history entries)

12.  API Response Format

All APIs return the same consistent response structure.

Success Response

{

"success": true,

"statusCode": 200,

"message": "Request created successfully",

"data": {},

"errors": null

}

Error Response

{

"success": false,

"statusCode": 400,

"message": "Validation failed",

"data": null,

"errors": ["Title is required", "Category is required"]

}

Backend C# Wrapper Class

public class ApiResponse<T>

{

public bool Success { get; set; }

public int StatusCode { get; set; }

public string Message { get; set; }

public T Data { get; set; }

public List<string> Errors { get; set; }

}
