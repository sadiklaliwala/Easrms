Employee Asset & Service Request Management System

Document Field

Details

Interns, Evaluators, Senior Tech Lead, Project Coordinator

.NET 8 Web API, EF Core/Dapper, SQL Server, React, TypeScript/JavaScript, Git

15 working days including planning, development, CR handling, testing and demo

Simulate a real-world project to evaluate development quality, ownership, estimation, communication and delivery behavior

Important Rule

Interns must prepare tasks, estimates, technical design, implementation, testing evidence and final demo artifacts.

1. Statement of Work (SOW)

The objective is to design and develop an internal enterprise-style portal where employees can raise asset and service requests, managers can approve eligible requests, admins can manage categories and assignments, and support users can resolve assigned requests. The project is intentionally scoped for a 15-day live project simulation but must be approached with real project discipline.

Scope Area

Description

Business Objective

Improve tracking of employee asset/service requests with transparent status, ownership and approval flow.

Primary Users

Employee, Manager, Admin, Support User

In Scope

Authentication simulation, role-based navigation, request creation, request listing/detail, category management, approval flow, support assignment, status tracking, dashboard, audit trail, CR implementation.

Out of Scope

Real SSO integration, production email/SMS, file storage in cloud, complex reporting engine, background workers unless implemented optionally.

Deliverables

Working React app, .NET API, SQL database schema/scripts, API collection, technical design notes, test cases, demo script, final code repository.

Constraints

Complete in 15 working days, follow Git discipline, submit impact analysis before implementing CR, demonstrate working flows.

2. SRS - Software Requirement Specification

Role

Permissions

Employee

Create request, view own requests, view request status and comments.

Manager

View team requests requiring approval, approve/reject, view dashboard summary.

Admin

Manage users/categories, assign support users, view all requests, update priority, access dashboard.

Support User

View assigned requests, update status, add resolution comments.

Functional Area

Requirement

Login & Roles

System should allow role-based access using seeded users or JWT-based authentication.

Category Management

Admin can create, edit, activate/deactivate request categories.

Request Creation

Employee can raise request with category, priority, description and optional attachment placeholder.

Approval Flow

Requests marked approval-required should go to manager before support assignment.

Assignment

Admin can assign approved/open requests to support users.

Status Tracking

Request status should move through Open, Pending Approval, Approved, Rejected, Assigned, In Progress, Resolved, Closed.

Comments

Manager/support/admin can add comments on request detail.

Dashboard

Role-based summary with request counts by status and priority.

Audit Trail

Important actions should be stored in status/history table.

Non-Functional Requirement

Expected Standard

Security

Role-based API authorization should be implemented or simulated cleanly. Sensitive data should not be exposed unnecessarily.

Validation

All required fields validated on frontend and backend.

Error Handling

Consistent error response model. No raw exception messages shown to users.

Performance

Listing APIs should support pagination/search/filter.

Maintainability

Layered backend and reusable React components.

Logging

API errors and important actions should be logged using a standard approach.

Testing

Interns should provide test cases and evidence for primary flows and CR regression.

5. TDD - Technical Design Document

Recommended backend structure: API Controllers, Application Services, Domain/Entities, Infrastructure/Repositories, DTOs/ViewModels, Validators, Middleware, Common response/error models. Recommended frontend structure: pages, components, services/api clients, hooks, context/store, routes, utils and validation schemas.

Layer/Area

Design Expectation

Backend Architecture

Layered .NET 8 Web API with controller-service-repository separation. Clean Architecture is preferred if interns are trained for it.

Authentication

JWT or simulated token-based login with role claims.

Database

SQL Server with normalized tables, FK relationships, audit fields, status history.

Validation

DTO-level validation and frontend validation.

Exception Handling

Global exception middleware returning common error response.

API Documentation

Swagger/OpenAPI enabled. API collection optional but recommended.

React Architecture

Route-based pages, reusable form/grid components, service layer for API calls.

State Management

Local state or context/store depending on complexity. Loading/error states mandatory.

Git

Feature branches, meaningful commits, PR/code review comments.
